## 2. 기획 의도

SOOP(구 아프리카TV) 소속 스트리머들이 참여하는 "LoL 멸망전" 이벤트에서, 스트리머들이 편리하게 팀을 구성하고, 시청자들이 경기 기록과 티어 정보를 손쉽게 확인할 수 있는 웹 서비스를 제공한다.

---

## 3. 문제점 분석 (As-Is)

- 스트리머들의 현재 롤 티어 정보를 한눈에 확인할 수 있는 공식 데이터베이스가 없음
- 팀 구성 과정이 수기/구두로 이루어져 라인 중복, 라인 불일치 등의 혼선이 발생하기 쉬움
- 대전 기록(전적)을 시청자가 확인할 수 있는 통합된 채널이 부재함

## 6. 사용자 시나리오

### 스트리머
- 기능 1 → 자신을 데이터베이스에 등록할 수 있다.
- 기능 2 → 자신의 팀을 구성하고 팀명을 정할 수 있다.
- 기능 3 → 자신의 팀의 대전 기록을 조회할 수 있다.

### 시청자
- 기능 1 → 데이터베이스 조회
- 기능 3 → 응원하는 팀의 대전 기록 조회

## 7. 서비스 플로우

### 7-1. 롤 티어 데이터베이스 (기능 1)

1. 스트리머가 데이터베이스에 미등록 상태인 경우, 등록 버튼을 클릭한다.
2. 스트리머명, 롤 아이디, 해시태그(태그라인)를 입력한다.
3. Riot API를 통해 전 시즌 티어/랭크 정보를 조회한다.
   - 전 시즌 데이터가 없으면 최근 전전 시즌 데이터를 조회한다 (참고용 데이터이므로 정합성보다 가용성 우선).
4. SOOP 스트리머 아이콘 이미지를 함께 가져온다.
5. `streamers` 테이블에 스트리머 아이콘, 스트리머 ID, 롤 ID, 롤 태그 정보를 저장한다 (seq가 PK).
6. `tiers` 테이블에 롤 티어, 랭크, 시즌 정보를 저장한다 (seq가 PK, streamer_seq가 FK).
7. 최초 페이지 로드 시 SSR로 데이터를 가져와 [참고 UI: cnine.kr/starcraft/tier] 형태의 페이지를 렌더링한다.
8. 상단에 티어별 바로가기 아이콘(챌린저/그랜드마스터 등)을 배치하여 해당 티어 구간으로 페이지 내 이동을 지원한다.

**인증 정책**: 누구나 무분별하게 데이터베이스에 등록하는 것을 방지하기 위해, "DB 등록하기" 버튼 클릭 시에만 SOOP OAuth 인증을 요구한다. 단순 데이터 조회는 인증 없이 가능하다.

> **확인 필요**: SOOP이 외부 개발자용 공식 OAuth API를 제공하는지 사전 조사 필요. 미제공 시 대체 인증 수단(채널 URL 인증 등) 검토.

### 7-2. 팀 빌드하기 (기능 2)

1. 팀장이 본인 확인(예: "이상호 확인")을 진행한다.
2. 확인 버튼 클릭 시 해당 스트리머가 자신의 포지션(라인)에 등록된다.
3. 상단 스트리머 풀(Pool)에서 아이콘을 드래그하여 팀을 구성한다.
4. 드래그한 스트리머의 실제 라인과 드롭 위치의 라인이 불일치할 경우, "해당 스트리머의 라인이 아닙니다"라는 경고 문구를 노출한다.
5. 라인 스왑 가능성을 고려하여, 경고 이후에도 "그래도 등록하시겠습니까?"라는 재확인 문구를 노출하고 강제 등록을 허용한다.
6. 팀 구성이 완료되면 "구성이 가능합니다 / 불가능합니다"로 결과를 정리한다.
7. "구성하시겠습니까?" alert 확인 후, `teams` 데이터에 팀이 최종 저장된다.
8. 구성된 팀 이미지를 캡처할 수 있는 기능을 제공한다 (시청자 공유용).

**저장 데이터 구조(개념)**: 팀명 / 팀장 / top·jungle·mid·ad·support (스트리머 ID) / 전적(승/패) / 승률(%)

### 7-3. 대전 기록 확인 (기능 3)

1. 팀장 본인 확인 후 진행 (또는 시청자는 조회만 가능).
2. "전체 / @@팀 / @@팀 / @@팀" 형태의 필터 버튼을 클릭한다.
3. 선택한 팀의 대전 기록만 필터링되어 노출된다.

## 8. 데이터베이스 설계

DB는 **PostgreSQL 단일**로 운영한다 (팀 라인업 등 유동적인 데이터는 JSONB 컬럼으로 유연성 확보).

### streamers
| 컬럼 | 타입 | 설명 |
|---|---|---|
| seq | PK | |
| streamer_name | varchar | 스트리머명 |
| streamer_id | varchar | 스트리머 계정 |
| streamer_icon_url | varchar | SOOP 프로필 이미지 |
| lol_id | varchar | 롤 게임 아이디 |
| lol_tag | varchar | 롤 태그라인 |
| soop_channel_id | varchar | SOOP 채널 식별자 |
| created_at | datetime | |

### tiers
| 컬럼 | 타입 | 설명 |
|---|---|---|
| seq | PK | |
| streamer_seq | FK → streamers | |
| lol_tier | varchar | 예: CHALLENGER, GRANDMASTER |
| lol_rank | varchar | 예: I, II, III, IV |
| lp | varchar | 예: 900, 1735 |
| season | varchar | 조회된 시즌 |
| fetched_at | datetime | API 조회 시각 |

### teams
| 컬럼 | 타입 | 설명 |
|---|---|---|
| seq | PK | |
| team_name | varchar | 팀명 |
| captain_streamer_seq | FK → streamers | 팀장 |
| lineup | jsonb | `{"top": streamer_seq, "jungle": ..., "mid": ..., "ad": ..., "support": ...}` |
| wins | int | 전체 승 |
| losses | int | 전체 패 |
| created_at | datetime | |

> 승률(%)은 wins/losses 기반으로 조회 시점에 계산하거나, 별도 컬럼으로 저장 후 갱신하는 방식 중 선택.

### matches

|---|---|---|

| team_b_seq | FK → teams | |

## 9. API 설계 (개발 우선순위 기준)

### Phase 1 — 스트리머 티어 DB
| Method | Endpoint | 설명 |
|---|---|---|
| POST | `/api/streamers` | 스트리머 등록 (닉네임/lolId/lolTag 입력 → Riot API 조회 → 저장) |
| GET | `/api/streamers` | 전체 조회 (티어순 정렬, 페이지네이션) |
| GET | `/api/streamers/{seq}` | 단일 조회 |
| GET | `/api/streamers?tier=CHALLENGER` | 티어별 필터 (상단 바로가기용) |
| GET | `/api/streamers?groupBy=tier` / `?groupBy=line` | 메인 페이지 "티어별/라인별" 토글용 그룹핑 조회 |
| GET | `/api/streamers?line=JGL&tier=GRANDMASTER` | 팀 구성 페이지 스트리머 풀 필터링 (라인+티어 조합 조회) |

### Phase 2 — 팀 구성
| Method | Endpoint | 설명 |
|---|---|---|
| POST | `/api/teams` | 팀 생성 (팀명, 팀장, 라인업). 라인 불일치 스트리머가 포함된 경우 `forceLineMismatch: true` 플래그로 강제 등록 허용 |
| GET | `/api/teams/{seq}` | 팀 상세 조회 |
| PATCH | `/api/teams/{seq}` | 라인업 수정 |
| GET | `/api/teams/validate-lineup?streamerSeq=&targetLine=` | 드래그앤드롭 시 실시간 라인 일치 여부 검증 (불일치 시 경고 문구 응답) |

### Phase 3 — 대전 기록
| Method | Endpoint | 설명 |
|---|---|---|
| POST | `/api/matches` | 경기 결과 등록 |
| GET | `/api/matches?teamSeq=` | 특정 팀 전적 조회 |
| GET | `/api/matches` | 전체 대진 조회 |

## 10. UI 와이어프레임

### 10-1. 스트리머 롤 티어 데이터베이스 (메인 페이지)

- 상단 타이틀: "SOOP 멸망전 스트리머 롤 티어 데이터베이스"
- **"티어별/라인별" 토글 스위치**: 동일 데이터를 티어 그룹 뷰 / 라인 그룹 뷰로 전환하여 표시
- **바로가기 아이콘**: 챌린저/그랜드마스터/마스터/다이아 등 티어 아이콘을 상단에 배치, 클릭 시 해당 티어 섹션으로 페이지 내 스크롤 이동
- **티어별 섹션 박스**: 챌린저 티어, 그랜드마스터 티어, 골드 티어 등 티어별로 구분된 박스 안에 스트리머 아이콘 나열
- 스트리머 아이콘에는 hover/클릭 시 상세 정보 카드 노출: 스트리머명 / LoL ID(아이디#태그) / 라인 / 랭크(예: 챌린저 1900, 그랜드마스터 800, 골드 4)

### 10-2. 팀 구성하기 페이지

- 상단 타이틀: "팀 구성하기", "팀 빌드" 버튼(아이콘)
- **좌측 필터 드롭다운**: 라인(TOP/JGL/MID/BOT/SPT), 티어(챌린저~아이언) — 조합하여 스트리머 풀을 좁혀서 탐색
- **스트리머 풀 영역**: 필터링된 스트리머 아이콘 목록을 카드 형태로 나열 (드래그 소스)
- **팀장 확인 영역**: "팀장: [닉네임]" + 확인 버튼 → 팀장 본인 인증
- **라인별 슬롯 테이블**: TOP/JGL/MID/BOT/SPT 5개 행, 각 행에 스트리머명 / 롤 닉네임 / 티어 표시. 풀에서 드래그하여 슬롯을 채움
- 드래그한 스트리머의 실제 라인과 슬롯 라인이 다르면 경고 문구 노출 → "그래도 등록하시겠습니까?" 재확인 후 강제 등록 가능 (7-2 플로우 참고)


### 10-3. 팀 구성 완료 확인 알럿

- "해당 팀은 구성이 가능합니다." 안내 문구
- 확인 / 취소 버튼
- 카메라(캡처) 아이콘: 구성된 팀 라인업 이미지를 캡처하여 시청자에게 공유 가능하도록 지원

### 10-4. 대결 기록 페이지

- 좌우 대비 구조: "[팀A명] [스코어, 예: 2:1] [팀B명]"
- 각 팀 하단에 라인별 슬롯 테이블(라인/스트리머명/롤 닉네임/티어)을 병기하여 어떤 매치업이었는지 한눈에 확인 가능
- 상단에 "전체 / 팀A / 팀B / 팀C ..." 필터 버튼으로 특정 팀의 대전 기록만 조회 가능 (7-3 플로우 참고)

## 11. 시스템 아키텍처 (개요)

```
[Client: React(Vite)]
        │
        ▼
[Spring Boot API Server] ──── [Riot API] (티어 조회)
        │              └────── [SOOP OAuth] (등록 시 인증, 조사 필요)
        ▼
[PostgreSQL] (streamers / tiers / teams(JSONB) / matches)
```

- 초기 단계는 단일 서버 + Docker 컨테이너로 구성, 이벤트 성격상 대규모 인프라(K8s 등)는 우선순위에서 제외
- 개발 순서: 백엔드(API/DB) → 프론트엔드(UI/UX) → 인프라(배포/모니터링)

---

## 12. 기술 스택 선정 이유

| 구성 | 선택 | 이유 |
|---|---|---|
| Frontend | React (Vite) | 팀 구성 드래그앤드롭 등 인터랙티브 UI 구현에 적합 |
| Backend | **Java Spring Boot** | 취업용 포트폴리오 목적상 채용 시장 수요가 높은 스택으로 선정 |
| DB | **PostgreSQL** (단일) | 관계형 데이터(스트리머-팀-경기) 처리에 적합, JSONB로 팀 라인업 등 유동 데이터 유연하게 저장 가능 |
| Infra | Docker (우선), 이후 확장 검토 | 이벤트성 서비스 특성상 초기엔 단일 서버로 충분, 추후 트래픽에 따라 확장 |

> 참고: DB는 최초 MariaDB+MongoDB 이원화안이 논의되었으나, 관리 복잡도 대비 실익이 적어 PostgreSQL 단일 + JSONB 컬럼 활용으로 단순화함.
